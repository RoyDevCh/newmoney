VoxYZ（voxyz.space）自动化深度调查报告
（撰写日期：2026-03-10，Asia/Taipei；研究范围：仅基于公开可访问页面、公开文档与公开代码仓库；不假
设任何私有 API 可用。）
执行摘要
VoxYZ（voxyz.space）对外呈现为一个“由 5 个 AI 代理运营的公司”，并将其背后的“角色分工 + 代码 + 指南”打
包销售，让用户在约 30 分钟内搭建自己的 AI 代理团队；其“Live Stage / Office / Radar”强调全流程可观测、
决策可追溯与持续运行。网站主页直接描述“5 AI Agents. One Company.”以及“Every decision is visible on a
live dashboard.”并将核心产品线分为开源的 Ship Faster、模板 Templates 与付费 Starter/Pro/咨询等。 
从自动化系统视角看，你的网站已经具备三个非常稀缺的资产：一是“公开可观测面”（Stage、Office）把运行
状态、交接、产物、甚至上下文窗口预算直接暴露出来；二是产品本身强调“可恢复、可审核、可审批的工作
流”（Ship Faster 的 runs/ 结构与 approval gates）；三是核心第三方链路（Stripe 支付、GitHub 私库交付、
Resend 邮件、Supabase 数据库/认证、Vercel 部署）已在隐私政策中明示。 
本报告的结论是：如果你的目标是“提升自动化系统的稳定性、可扩展性与可被外部工具（RPA/CI）可靠集
成”，优先级最高的改造并不是再加更多 UI，而是把现有的“公开可观测面”升级为“稳定的机器接口（事件流 +
可查询 API + 明确的鉴权与限流）”，并将支付/交付链路（Stripe→验证→GitHub 邀请→邮件通知）做成可重
放、可审计、可告警的闭环。 
站点定位、目标用户与关键功能
站点目的与商业包装
VoxYZ  的对外叙事是“5  个代理在做真实工作”，包含研究需求、策略辩论、内容撰写、产品交付，并把“完成这
套系统所需的角色、代码和指南”打包，让用户复制同样的组织形态。主页明确把 판매（What  We  Sell）定义
为“完整系统 behind  this  AI  company”，并区分免费开源组件（Ship  Faster、Templates）与付费 Starter/
Pro/咨询。 
“About”页进一步强化“无假职位（No  Fake  Jobs）”“公开构建（Built  in  Public）”“24/7  持续运行（Always
Running）”，并展示代理之间关系矩阵与最近“关系变化”的解释文本，属于“把内部行为当作产品卖点”的典型模
式。 
目标用户画像（以可自动化为中心）
从主页与 Ship Faster 仓库的描述综合判断，核心目标用户主要是：
1）使用 AI 编码/代理工具的开发者与独立构建者（Ship Faster 明确提到 Claude Code、Cursor、OpenCode
等，以及“skills  是主要接口”）；2）希望“可复现、可审计、可恢复”的团队操作者；3）缺乏编码背景但希望“复
制一套可运行的代理组织”的购买者（主页评价区有“no coding skills”“2 hours setup”等反馈）。 
对自动化系统而言，这意味着：用户会强烈依赖“确定性输出、状态可见、失败可恢复”，并且非常在意“交付/安
装/运行”的自动化程度（尤其是 Starter 所承诺的“30 分钟上手、无编码”）。

自动化接口与可扩展面
已确认的外部依赖与“事实接口”
你的隐私政策列出核心第三方：Stripe（支付）、GitHub（仓库邀请交付）、Supabase（数据库与认证）、
Resend（邮件）、Vercel（托管与 analytics）、Telegram（买家社群）。这些第三方本身都具备成熟自动化
接口（Webhook/API/SDK），即便你没有公开“VoxYZ  API”，也已经天然存在一条“可自动化的业务控制面”。
此外，Office 页面直接暴露了更接近“内部事件系统”的线索：
-  Shipping  Log  中的每个产物带有 “Deploy”  链接，指向 *.vercel.app，说明交付/演示应用部署在
Vercel，并且“从 signal 到 shipped”的时间戳与置信度被结构化展示。 
- “VPS Context Doctor”宣称是“Public snapshot of the live VPS OpenClaw workspaces”，并展示 token
budget、workspace  files、installed  skills，以及“Published  from  the  live  VPS  observer  into
ops_agent_events.”——这几乎等同于你在公开页面承认存在一个（至少内部）事件表/事件流。 
这两点非常关键：它们意味着你已经拥有“事件源（agent  runtime/VPS  observer）→ 存储（可能是
Supabase）→ 展示（Office/Stage）”的链路，只差把它“稳定化、文档化、对外可控地开放”。

公开页面中可见的“自动化表面”
Stage  页面展示“Live  Feed  /  Tasks  /  Social  /  Dashboard”，并以分钟级/秒级节奏滚动；每个代理旁边显示
model 与 role（例如 Nexus: Claude Opus 4.6 Coordinator；Scout: GPT-5.2 Codex Research 等）。这类 UI
通常背后至少需要一个 polling 或推送机制，但页面未直接公开接口。 
Office  页面包含更丰富的结构化实体：Signals、Shipped  items、War  Rooms、Timesheets、Handoffs、
Accuracy  Leaderboard、以及“workspace  snapshot（tokens/free/budget/files/skills）”。即使你不打算公
开写接口，也强烈建议公开只读接口（或导出）来支持外部自动化（例如：自动生成周报/战情室摘要/版本发布
说明）。 
Radar  页面在本次抓取中返回空内容（0  行），这会直接阻断搜索引擎与 RPA/无头浏览器的“低成本读取”，也
让外部自动化难以稳定对接。你需要决定：Radar  是否应当公开只读数据（例如匿名可看摘要、登录可看细
节），并提供可机器读取的“分页 API / 导出”作为替代。

数据层、认证与邮件
隐私政策明确：Supabase 用于数据库与认证、Resend 用于邮件、Stripe 用于支付。对自动化系统而言，这几
乎天然等同于：
- 身份/会话：Supabase Auth（cookie/local storage 会话也在隐私政策中出现）；
- 交易事实：Stripe（订单、支付状态）；
- 交付控制面：GitHub repo 邀请；
- 通知：Resend 邮件。 
“代理运行时”与可观测链路
Office  页面“VPS  Context  Doctor”是最接近“内部运行时架构剖面”的证据：它声称自己在展示 “live  VPS
OpenClaw workspaces”，并公开 token budget、workspace files、installed skills；同时提到从 “live VPS
observer”  发布到 “ops_agent_events”。这说明你可能存在一个常驻 VPS/服务器上的代理运行时与观察者/采
集器组件。 
OpenClaw  文档描述其作为“self-hosted  gateway”，提供多渠道入口（WhatsApp/Telegram/Discord/
iMessage  等）、会话和路由单一真相源（single  source  of  truth），并提供 Web  Control  UI、配置文件、
allowlist 等安全控制；这很符合“代理组织的运营入口”与“移动端随时触达”的定位。 
综合上述公开信息，可以给出一个“高概率”架构推断：Vercel/Next.js  承载展示层与部分 API  glue；Supabase
作为数据存储与认证；Stripe/GitHub/Resend 构成交付闭环；VPS 上运行代理与 OpenClaw gateway，并持续
产出事件写入数据层，再由 Office/Stage  消费展示。该推断的直接证据在于：Office  明示 VPS  snapshot  与
ops_agent_events、隐私政策列出 Supabase/Vercel、条款列出 GitHub 私库交付。

安全与隐私态势
隐私政策与数据边界
隐私政策明确采集项包括：订阅/购买邮箱、Stripe 订单元数据（不存储完整卡号）、GitHub username（用于

常见漏洞面与建议（以“自动化流程”优先）
由于本次研究未直接获得站点 headers/CSP 配置与速率限制细节（未发现可引用的公开扫描结果），以下建议
